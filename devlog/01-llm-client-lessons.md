## Хронология ошибок и исправлений

### Ошибка 1 — Неправильный endpoint (HTTP 400/502 при первом запуске)

**Симптом:** `llm_unavailable: HTTP 400`, затем `HTTP 502`.

**Причина:** в `NewLLMClient` дефолтный `baseURL` для Anthropic был задан как
`"https://api.anthropic.com"` без суффикса `/v1`. В итоге запрос уходил на
`POST https://api.anthropic.com/chat/completions` — несуществующий путь.

Правильный endpoint: `POST https://api.anthropic.com/v1/chat/completions`.

**Исправление:** добавить `/v1` в дефолтный `baseURL`:
```go
// было
baseURL = "https://api.anthropic.com"
// стало
baseURL = "https://api.anthropic.com/v1"
```

**Инженерная недоработка:** endpoint был заложен в конструктор без проверки
на реальной модели. Отсутствие интеграционного теста (хотя бы ручного)
позволило дефекту дойти до первого запуска.

---

### Ошибка 2 — Бюджет токенов занижен в 4-5 раз (llm_budget_exceeded)

**Симптом:** `llm_budget_exceeded: usage 233973 > 50000` — первый же вызов.

**Причина:** константа `tokenBudgetLimit = 50_000` была взята «с потолка» без
анализа реального размера документации целевых репозиториев. Репа passkey-demo-api
содержит 851 KB markdown-файлов; даже три doc-файла дают ~30–50k токенов
входного контекста плюс ответ модели.

**Исправление:** поднять лимит до `300_000`. Позже проблема решилась
принципиально через фильтрацию docs (ошибка 3).

**Инженерная недоработка:** `tokenBudgetLimit` — защитный предохранитель от
runaway-вызовов, а не проектный параметр. Он должен быть выше реального
максимума, не ниже него. Значение не было откалибровано по целевым репозиториям.

---

### Ошибка 3 — Весь корпус документов в каждый промпт (rate limit + огромный контекст)

**Симптом:** после поднятия `tokenBudgetLimit` первый вызов проходил,
второй падал с `llm_rate_limited` (429). Перезапуски не помогали —
rate limit не сбрасывался.

**Причина:** `RepoStore.ReadMarkdownDocs` читал **все** `.md`-файлы репозитория
рекурсивно. Для passkey-demo-api это 30+ файлов, 851 KB, ~233k токенов.
Каждый из четырёх JTBD-промптов получал весь корпус. Суммарно за одну
команду `fitness` улетало ~900k токенов — в 9× выше TPM-лимита tier-1.

После первого вызова (233k токенов) TPM-окно было заполнено полностью,
второй вызов немедленно получал 429. Никакая пауза 10–30 секунд не помогала:
окно в 1 минуту было заблокировано одним вызовом.

**Исправление:** вынести список проверяемых doc-файлов в конфиг (`docs:`),
добавить `ReadMarkdownDocsByList` — читает только указанные файлы,
пропускает отсутствующие. Дефолт: `[README.md, CLAUDE.md, CONTRIBUTING.md, AGENTS.md]`.

```yaml
docs:
  - README.md
  - CLAUDE.md
  - CONTRIBUTING.md
  - AGENTS.md
```

**Инженерная недоработка:** отправка «всего репозитория» в LLM — типичная
ошибка первого прототипа. Для L5 (JTBD-оценка документации) нужны конкретные
doc-файлы, не весь контент. Ни задача, ни дизайн не содержали явного решения
о том, что именно отправлять — это должно быть принято на этапе проектирования
и зафиксировано в конфиге.

---

### Ошибка 4 — Rate limit от последовательных вызовов без паузы

**Симптом:** даже после введения фильтра docs (4 файла, ~15k токенов/вызов)
второй или третий вызов из четырёх иногда получал 429.

**Причина:** четыре вызова `Simulate` шли один за другим без задержки.
На tier-1 Anthropic API ~60 запросов в минуту, но **токены в минуту (TPM)**
ограничены отдельно. При 15k токенов/вызов × 4 роли = 60k токенов
за ~2-3 секунды — легко превышает TPM-лимит.

**Исправление:** пауза 10 секунд между вызовами:
```go
for i, p := range set.Prompts() {
    if i > 0 {
        time.Sleep(10 * time.Second)
    }
    ...
}
```

**Инженерная недоработка:** последовательные LLM-вызовы в цикле без паузы —
ошибка проектирования I/O-объекта. Rate limiting — документированное поведение
любого LLM API. `Simulate` должен был учитывать это изначально.

---

### Ошибка 5 — Markdown-обёртка JSON несмотря на инструкцию «только JSON»

**Симптом:** `parse verdict: invalid character '`' looking for beginning of value` —
модель возвращала ` ```json\n{...}\n``` ` вместо голого JSON.

**Причина:** инструкция в промпте «Верни ТОЛЬКО JSON без пояснений» —
**не гарантия**. Модель её игнорирует в ~30–50% случаев в зависимости
от объёма контекста и версии модели.

**Исправление первого прохода:** `extractJSON` — вырезает первый `{...}`
из любого ответа:
```go
func extractJSON(s string) string {
    start := strings.Index(s, "{")
    end := strings.LastIndex(s, "}")
    ...
}
```

**Правильное исправление:** `response_format` со схемой вердикта —
структурированный вывод на уровне протокола (ошибка 6).

**Инженерная недоработка:** полагаться на промпт для управления форматом
вывода — антипаттерн. Формат вывода должен быть задан через API-механизм,
не через инструкцию в тексте.

---

### Ошибка 6 — Неправильный тип `response_format` (три итерации)

После решения использовать structured output выяснилось три уровня несоответствия
между кодом и документацией Anthropic:

**Итерация 1:** `"type": "json_object"` → HTTP 400:
```
Input should be 'json_schema'
```
Anthropic OpenAI-слой не поддерживает `json_object`, только `json_schema`.

**Итерация 2:** `json_schema` без `strict` → HTTP 400:
```
response_format.json_schema.strict: Field required
```

**Итерация 3:** `json_schema` + `strict: true` без `additionalProperties: false`
→ HTTP 400 при некоторых схемах.

**Финальный рабочий формат:**
```json
{
  "type": "json_schema",
  "json_schema": {
    "name": "verdict",
    "strict": true,
    "schema": {
      "type": "object",
      "properties": { ... },
      "required": [...],
      "additionalProperties": false
    }
  }
}
```

**Инженерная недоработка:** три итерации курл-диагностики по одному
неизвестному параметру — признак того, что `response_format` не был
проверен до написания кода. Нужно было сначала убедиться curl-ом,
что формат работает, и только потом кодировать структуру запроса.
Это стандартная практика при работе с любым внешним API.

## Принципы, выкристаллизованные в итерациях

**P1. Проверяй endpoint curl-ом до написания кода.**
Любой новый LLM-провайдер или новый тип запроса — сначала минимальный
curl, потом конструктор. Три ошибки из шести были обнаружены именно curl-диагностикой.

**P2. Формат вывода — через API-механизм, не через промпт.**
`response_format` с JSON-схемой — единственная гарантия структурированного ответа.
Инструкция «верни только JSON» — подсказка модели, не контракт.
`extractJSON` оставляем как второй эшелон защиты.

**P3. Граница контекста — проектное решение, не дефолт «всё».**
Список doc-файлов принадлежит конфигу. «Отправить весь репозиторий» —
не решение, а отложенная авария. Нужно явно ответить на вопрос:
«какие именно документы оцениваем?» — до первой строчки I/O-кода.

**P4. Rate limiting — не edge case, а штатный сценарий.**
Последовательные LLM-вызовы в цикле требуют паузы. Количество вызовов
и размер каждого должны умещаться в TPM-окно провайдера. Это считается
на этапе дизайна, не после первого 429.

**P5. Защитный лимит токенов — выше реального максимума, не ниже.**
`tokenBudgetLimit` — предохранитель от аномалий, не ограничение нормальной
работы. Калибруется по реальным репозиториям.

## Что передать Opus для проработки скилла

1. **Раздел «Верификация endpoint до кода»** — добавить в скилл явный шаг:
   перед реализацией LLM-клиента проверить curl-ом минимальный запрос
   к провайдеру с нужным `response_format`. Включить канонные примеры
   curl для Anthropic OpenAI-слоя.

2. **Раздел «Граница контекста»** — добавить формулу оценки токенов:
   `(размер docs в байтах / 4) × N_ролей < TPM_лимит`. Дефолтный список
   doc-файлов в конфиге обязателен.

3. **Раздел «Rate limiting»** — добавить таблицу: TPM-лимиты основных тиров
   Anthropic, формула паузы между вызовами, рекомендованные значения.

4. **Раздел «response_format»** — добавить полную рабочую curl-команду
   с `json_schema + strict + additionalProperties:false` как baseline.
   Указать явно: `json_object` Anthropic не поддерживает.

5. **Чеклист «до первого запуска»** — отдельный блок: endpoint verified,
   auth verified, response_format verified, docs list defined,
   token budget calibrated, rate limit accounted.
