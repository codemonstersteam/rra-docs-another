# rra-docs-another — концепция

> Универсальный аудитор качества документации **произвольного** git-репозитория.
> Отвечает на вопрос: **«закроет ли эта документация работу каждого из четырёх
> потребителей?»** — на любом репо, без предположений о его конвенциях.

Часть продукта **Rational Repository Auditor (RRA)**, но по позиционированию —
самостоятельный инструмент оценки качества. В отличие от узкого гейта `rra-docs`
(который проверяет соответствие дисциплине из скиллов), `rra-docs-another`
**не предполагает** рациональную дисциплину и работает на любом репозитории с
гитхаба. Это более универсальная и более сложная проверка.

## Источники

- https://codemonsters.team/blog/2026/04/07/documentation-as-product-jtbd/ — «документация = продукт с 4 JTBD-потребителями»
- https://codemonsters.team/blog/2026/05/02/rational-design-skills/

Нормативный источник для слоёв L1–L3 (формулы читаемости, prose-линтеры,
инструменты) — [`docs/concept-docs-readability.md`](./docs/concept-docs-readability.md).

## Позиционирование

`rra-docs-another` — **не** линтер орфографии, **не** счётчик покрытия и **не**
проверка дисциплины. Он измеряет **пригодность документации для работы (job)**
каждого из четырёх потребителей из статьи «Документация как продукт»:

| Потребитель | Job (работа, на которую «нанимают» документацию) |
|---|---|
| Инженер-создатель | Сдать репозиторий преемнику без личного брифинга |
| Инженер-потребитель | За 3 минуты оценить пригодность сервиса |
| Менеджер | За 3 минуты понять структуру системы |
| ИИ-агент | Загрузить контекст для конкретной задачи в ограниченном бюджете |

**Итог не усредняется.** «Документация для всех = ни для кого» — на выходе четыре
независимых score, по одному на потребителя, а не одна цифра.

## Принцип «дешёвая модель применяет, дорогая думает»

Оценка слоистая. ИИ привлекается **только на верхних слоях** и **только если**
нижние пройдены — статика и линтеры дают ~90% сигнала бесплатно.

| Слой | Отвечает на вопрос | Инструмент | ИИ | Интеграция |
|---|---|---|---|---|
| **L1. Читаемость** | «написано ли понятно?» | Flesch-Kincaid, формула Оборневой (рус.) | нет | — (чистая логика) |
| **L2. Стиль** | «нет ли когнитивного шума?» | Vale, markdownlint-cli2 | нет | subprocess |
| **L3. Структурная полнота** | «есть ли базовые файлы и связи?» | обход ФС | нет | ФС |
| **L4. JTBD-присутствие** | «есть ли разделы под job каждого потребителя?» | статический чеклист по словарю заголовков | нет | ФС |
| **L5. JTBD-пригодность** | «закроет ли содержимое job за бюджет?» | LLM-симуляция 4 ролей | да | Anthropic API |
| **L6. Согласованность с репозиторием** | «дока не разошлась с реальностью?» | извлечь проверяемые утверждения → сверить с ФС/кодом; LLM опционально | нет (тир L6c опц.) | ФС (+ опц. Anthropic API) |

L1, L3, L4 и ядро L6 — чистая логика над данными с ФС: ноль интеграций. L2 шеллит
во внешние линтеры. В LLM ходит только L5 и опциональный тир L6c.

### L1. Читаемость

Формулы readability реализуются **нативно на Go**: Flesch-Kincaid Grade для
английского, формула Оборневой `FRE = 206.836 − 1.52·ASL − 65.14·ASW` для
русского (слогоделение ≈ счёт гласных). `textstat` (Python) — опциональная сверка
при разработке, не runtime-зависимость. L1 — порог, а не приговор: формулы ловят
поверхностные признаки (термин `WebAuthn` посчитают «сложным»), поэтому
проваленный L1 даёт warning, не блок.

### L2. Стиль

Через subprocess: **Vale** (prose-линтинг, кастомные правила терминологии и длины
предложений) и **markdownlint-cli2** (структура markdown). Профиль правил
переопределяется на проект.

### L3. Структурная полнота (универсальная)

Go-обход репозитория без предположений о дисциплине: есть ли `README`, резолвятся
ли ссылки между документами, drift (`docs/` старше кода на > N дней),
дублирование фрагментов. Проверка наличия **конкретных** дисциплинарных файлов
(`intent.md`, `messages.md`, `contracts-graph.md`) сюда **не** входит — это задача
гейта `rra-docs`.

### L4. JTBD-присутствие (статика)

Для каждого из четырёх потребителей — детерминированный чеклист: какие секции
**обязаны существовать**, чтобы job был выполним. Секции ищутся matching-ем
нормализованных заголовков H1/H2/H3 против словаря синонимов (русский +
английский, конфигурируется на проект). Выход — четыре чек-карточки
`PASS/FAIL/PARTIAL` с `file:line`. Провалить L4 и пройти L5 — нельзя.

### L5. JTBD-пригодность (LLM)

Запускается только если L4 не упал в `FAIL`. Четыре независимых прогона:

| Потребитель | Бюджет | Критерий успеха |
|---|---|---|
| Инженер-создатель | `docs/` целиком | LLM отвечает на контрольные вопросы с цитатами |
| Инженер-потребитель | ~2000 токенов README + 1 файл | LLM формулирует: что делает, интеграции, лимиты |
| Менеджер | ~1500 токенов | LLM перечисляет подсистемы и их назначение |
| ИИ-агент | ~8000 токенов | LLM находит нужные файлы и формулирует план изменения |

Каждый прогон — бинарный исход + список конкретных пробелов.

### L6. Согласованность с репозиторием

L6 спрашивает: **документация утверждает то, что репозиторий опровергает?** Это
«дрейф документации», проверяемый на любой репе без привязки к конвенциям.

**L6a — универсальный дрейф (детерминированно, без ИИ).** Берём только
утверждения, проверяемые механически по ФС/коду:

| Утверждение в доке | Проверка по репозиторию |
|---|---|
| относительные ссылки и пути в backticks | резолвятся в ФС? битая = дрейф |
| fenced-команды (` ```bash … ``` `) | референс существует (`make <target>`, `go run ./cmd/x`, compose)? |
| «Стек: Postgres, Redis…» | зависимость есть в `go.mod` / `package.json` / `requirements`? |
| env-переменные / config-ключи | grep по коду находит? |
| список CLI-подкоманд / эндпоинтов | зарегистрированы в коде / в OpenAPI? |

Это grep + парсинг манифестов. Нарушение заявляется, только если подтверждается
механически.

**L6c — семантический тир (опционально, LLM).** Утверждения, которые grep не
берёт («README обещает ретрай на 503 с backoff»), проверяет LLM строго
ограниченно: либо извлекает утверждение в машинную форму (дальше проверка
механическая), либо судит конкретную предъявленную пару (сниппет доки + кусок
кода) → Y/N + цитата. Запрещено «прочитай весь репозиторий». Включается флагом.

> Дисциплина-сверка `intent`/`messages`/`contracts-graph` (бывший L6b) сюда
> **не** входит — она в гейте `rra-docs`, поскольку специфична для дисциплины.

## Продуктовая поверхность

CLI (имя бинаря — рабочее, может быть укорочено):

```
rra-docs-another structure   <path>   # L3
rra-docs-another readability <path>   # L1
rra-docs-another jtbd        <path>   # L4
rra-docs-another style       <path>   # L2
rra-docs-another fitness     <path>   # L5  (LLM)
rra-docs-another drift       <path>   # L6  (ядро без ИИ; --semantic — LLM)
rra-docs-another assess      <path>   # весь пайплайн, дешёвое-первым, short-circuit
```

Флаги: `--format md|json`, `--up-to L1..L6`, `--config <file>`, `--out <file|->`.
**Выход** — Markdown + JSON по `report.schema.json`. **Коды возврата**: `0` — все
JTBD прошли; `1` — есть проваленный JTBD; `2` — ошибка запуска.

## I/O-граница

Спроектирован по той же дисциплине, что измеряет: логика — чистые функции, внешний
мир — в автономных I/O-объектах.

| I/O-объект | Зависимость (скрыта внутри) | Используют слои |
|---|---|---|
| `RepoStore` | ФС / git (чтение `.md`, манифестов, кода) | все |
| `LinterRunner` | subprocess Vale / markdownlint | L2 |
| `LLMClient` | Anthropic API | L5 (+ L6c опц.) |
| `ReportSink` | stdout / файл | все |

## Связь с другими репозиториями RRA

Все репозитории — под [github.com/codemonstersteam](https://github.com/codemonstersteam/):

- **`rationaldev`** — зонтичная концепция RRA, оба критерия, стратегия итераций.
- **`rra-docs`** — узкий **гейт соответствия дисциплине** (детерминированно, без
  LLM); применяется в CI на дисциплинарных проектах. Содержит бывший L6b.
  [github.com/codemonstersteam/rra-docs](https://github.com/codemonstersteam/rra-docs)
- **`rra-audit-repo`** — критерий №1 (I/O-граница по языкам) + точка входа.
  [github.com/codemonstersteam/rra-audit-repo](https://github.com/codemonstersteam/rra-audit-repo)
- **`rra-docs-another`** (этот репозиторий) — универсальная оценка качества
  документации произвольных репо.
