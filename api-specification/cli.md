# Контракт CLI — rra-docs-another

> Источник истины об интерфейсе. Для CLI роль OpenAPI играет этот файл +
> [`report.schema.json`](./report.schema.json) (схема машинного отчёта). Изменения
> интерфейса без изменения контракта недопустимы (AGENTS.md §2).

`rra-docs-another` — универсальный аудитор качества документации произвольного
git-репозитория. Семь подкоманд, по одной на внешний вход (slice). Каждая
печатает отчёт по [`report.schema.json`](./report.schema.json) (JSON) или Markdown.

## Синопсис

```
rra-docs-another <команда> [path] [флаги]
```

- `<команда>` — одна из семи (см. ниже). Обязательна.
- `[path]` — путь к корню репозитория. Необязателен, по умолчанию `.`
  (текущая директория).

Без команды или с неизвестной командой — usage в stderr, код возврата `2`.
`version` / `--version` печатает версию (код `0`). `help` / `--help` печатает
usage (код `0`).

## Команды

| Команда | Слой | Вопрос | LLM |
|---|---|---|---|
| `structure` | L3 | есть ли базовые файлы и резолвятся ли связи? | нет |
| `readability` | L1 | написано ли понятно? | нет |
| `jtbd` | L4 | есть ли разделы под job каждого потребителя? | нет |
| `style` | L2 | нет ли когнитивного шума? | нет |
| `fitness` | L5 | закроет ли содержимое job за бюджет? | **да** |
| `drift` | L6 | дока не разошлась с репозиторием? | нет (с `--semantic` — да) |
| `assess` | L1–L6 | весь пайплайн, дешёвое-первым, short-circuit | да (если дошло до L5) |

`assess` исполняет слои по возрастанию стоимости и применяет short-circuit:
если L4 (`jtbd`) упал в `FAIL`, L5 (`fitness`, LLM) не запускается — деньги на
LLM не тратятся там, где дёшево уже видно провал. JTBD не усредняется: на выходе
четыре независимых результата.

## Флаги

### Общие (все команды)

| Флаг | Значение | По умолчанию | Описание |
|---|---|---|---|
| `--format` | `md` \| `json` | `md` | формат отчёта: Markdown для человека, JSON по схеме для CI |
| `--out` | `<файл>` \| `-` | `-` | куда писать отчёт; `-` — stdout |
| `--config` | `<файл.yml>` | вшитый дефолт | проектный конфиг (YAML): словари заголовков L4, профиль линтеров L2, пороги, `llm`-подключение и `prompts` ролей L5. Без секретов — ключ через `llm.api_key_env`. См. `docs/adr/0003-yaml-config.md` |

### LLM (значимы для `fitness`, `assess`, `drift --semantic`)

| Флаг | Значение | По умолчанию | Описание |
|---|---|---|---|
| `--llm-provider` | `anthropic` \| `openai` | `anthropic` | провайдер LLM (см. `LLMClient`, CONCEPT §I/O-граница) |
| `--llm-base-url` | `<url>` | — | базовый URL OpenAI-совместимого эндпоинта (OpenAI, Azure, OpenRouter, Ollama, vLLM…); только при `--llm-provider openai` |
| `--llm-model` | `<model>` | провайдер-специфичный | имя модели |

Ключ API — из окружения, не из флага: `ANTHROPIC_API_KEY` (провайдер `anthropic`)
или `OPENAI_API_KEY` (провайдер `openai`). Отсутствие ключа на команде, которой
он нужен → `error.code=llm_unavailable`, код возврата `2`.

### Специфичные

| Команда | Флаг | Значение | По умолчанию | Описание |
|---|---|---|---|---|
| `assess` | `--up-to` | `L1`..`L6` | `L6` | остановить пайплайн на указанном слое включительно |
| `drift` | `--semantic` | — (булев) | выкл. | включить семантический тир L6c (LLM-сверка предъявленных пар) |

## Коды возврата

| Код | Значение |
|---|---|
| `0` | команда отработала, блокирующих провалов нет |
| `1` | команда отработала, есть блокирующий провал (см. ниже) |
| `2` | ошибка запуска: оценка не выполнена (ввод/интеграция, `error.code` см. README «Карта режимов отказа») |

Что считается **блокирующим провалом** (код `1`) по командам:

- `jtbd`, `fitness`, `assess` — хотя бы один JTBD-результат в статусе `FAIL`.
- `structure`, `style`, `drift` — хотя бы одно нарушение с `severity=blocker`.
- `readability` — **никогда** не даёт код `1`: L1 — порог-warning, не блок
  (CONCEPT §L1); низкая читаемость отражается в `score`/`violations`, но код `0`.

Код `2` отличается от `1` тем, что оценка **не выполнена** (репозиторий не
прочитан, линтер не найден, LLM недоступен), а не «выполнена и нашла проблему».
Один `error.code` ⇒ всегда код `2`; различаются `error.code` и действием
пользователя (полная таблица — README «Карта режимов отказа»).

## Вывод

- `--format md` — человекочитаемый Markdown-отчёт в `--out`.
- `--format json` — объект по [`report.schema.json`](./report.schema.json) в `--out`.
- Диагностика и `error.code` (код `2`) — в stderr; в `--format json` ошибка
  дополнительно попадает в `errors[]` отчёта.

Послойные команды заполняют свою секцию `layers`; `jtbd`/`fitness`/`assess`
заполняют секцию `jtbd` (четыре потребителя). Поле `command` отчёта фиксирует,
какая подкоманда его произвела.

## Примеры

```sh
# Структурная полнота, человекочитаемо
rra-docs-another structure ./my-repo

# Полный аудит в JSON для CI, до L4 (без LLM)
rra-docs-another assess ./my-repo --format json --up-to L4 --out report.json

# JTBD-пригодность через локальный OpenAI-совместимый эндпоинт
OPENAI_API_KEY=sk-… rra-docs-another fitness ./my-repo \
  --llm-provider openai --llm-base-url http://localhost:11434/v1 --llm-model qwen2.5
```
