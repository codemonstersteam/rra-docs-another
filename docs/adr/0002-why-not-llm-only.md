# 0002 — Почему не LLM-only: ПО как слоистый детерминированный харнесс вокруг LLM

## Статус

accepted

## Контекст

Закономерный вопрос к самому существованию проекта: зачем писать ПО (слои L1–L6,
CLI, I/O-объекты), если качество документации можно «просто проверить моделью» —
скормить репозиторий LLM и попросить оценку?

`rra-docs-another` обещает не разовое мнение, а **продукт** (см. `intent.md`):
четыре независимых score по четырём JTBD-потребителям плюс список конкретных
пробелов, воспроизводимо, на произвольном репо. Под это обещание «спросить
модель» ломается по четырём осям:

- **Воспроизводимость.** Один репо → один результат, иначе число нельзя ставить в
  CL/CI и нельзя верить ему дважды. LLM на одном промпте даёт разброс прогон от
  прогона. L1–L4 и ядро L6 детерминированы по построению.
- **Цена и охват.** Гонять LLM по всему репо на каждый коммит — дорого и медленно.
  Статика и линтеры дают ~90% сигнала бесплатно (CONCEPT.md, принцип «дешёвая
  модель применяет, дорогая думает»). Модель вызывается только если нижние слои
  прошли (short-circuit), и только с узким срезом — «прочитай весь репозиторий»
  запрещено даже на L6c.
- **Объективное нельзя отдавать на суждение.** Flesch-Kincaid, резолвятся ли
  ссылки, есть ли зависимость из «Стек: …» в `go.mod`, существует ли `make`-таргет
  из fenced-команды (L6a) — это арифметика и парсинг. Спрашивать про это модель —
  дороже и **менее** точно, чем grep; модель тут источник галлюцинаций, а не
  помощник.
- **Аудируемость.** «Битая ссылка в `README.md:42`» — действие. «Модель оценила на
  7/10» — ничего. L4 даёт чек-карточки PASS/FAIL/PARTIAL с `file:line`.

Голый вызов LLM вдобавок схлопывает четырёх потребителей в одну кашу, а концепт
прямо против этого: «документация для всех = ни для кого» (CONCEPT.md).

При этом обратное тоже верно: для разового «нормальный ли тут README?» ПО не
нужно — модель и сама справится. Водораздел — в обещании воспроизводимого,
структурированного, грунтованного результата, а не в наличии суждения как
такового.

Рассмотренные альтернативы:

- **LLM-only (один промпт на весь репо).** Невоспроизводимо, дорого на охвате,
  не грунтовано до `file:line`, не даёт раздельных score. Отвергнуто как продукт
  (годится только для ad-hoc вопроса).
- **Детерминированно-only (без LLM вообще).** Берёт измеримое (L1–L4, L6a), но не
  закрывает суждение «закроет ли содержимое job за бюджет» (L5) и семантический
  дрейф (L6c). Это уже соседний гейт `rra-docs`, а не данный продукт.

## Решение

Строим ПО как **слоистый детерминированный харнесс, который оркеструет LLM**, а
не заменяется им. Модель — один компонент из шести, запертый туда, где суждение
неизбежно (L5 JTBD-пригодность, опциональный L6c), и кормится узким срезом.
Ценность ПО — ровно та часть, которая **не** модель: детерминизм, дешевизна,
объективность измеримого, ссылки на строки и решение «когда и с каким срезом идти
в LLM».

## Последствия

**Проще:**
- Результат воспроизводим и аудируем (`file:line`), пригоден для CI и кодов
  возврата.
- LLM-вызовы дёшевы и редки: только после прохождения нижних слоёв и только узким
  срезом.

**Сложнее:**
- Нужно поддерживать детерминированные слои (словари заголовков L4, парсеры
  манифестов L6a) — это код, который при LLM-only «как бы» не писали (но
  получали бы невоспроизводимость взамен).
- Граница «что считать объективным (греп) vs суждением (LLM)» требует дисциплины:
  соблазн отдать пограничный случай модели всегда есть.

**Триггер пересмотра:** если стоимость и латентность LLM упадут настолько, что
воспроизводимость станет достижима иначе (например, через закреплённый seed +
кэш + структурированный детерминированный вывод модели), и при этом охват
всего репо станет дешёвым — вернуться к балансу слоёв.
