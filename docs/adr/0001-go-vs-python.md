# 0001 — Язык реализации rra-docs-another: Go (+ при необходимости Python-сайдкар на L1)

## Статус

accepted

## Контекст

`rra-docs-another` — универсальная оценка качества документации произвольных
репо (слои L1–L6). Встал вопрос: насколько рационально реализовывать его на Go,
учитывая, что соседние репо RRA (`rra-docs`, `rra-audit-repo`) уже на Go.

Ключевые наблюдения:

- **Тяжёлая семантика делегирована LLM** (L5 JTBD-симуляция, L6c), а не локальному
  NLP. Главный довод за Python — зрелая NLP-экосистема (spaCy, NLTK, эмбеддинги) —
  здесь почти не работает: «думает» язык-агностичный LLM.
- Локально остаются только дешёвые детерминированные слои:
  - **L1 (читаемость)** — единственная зона объективного преимущества Python
    (`textstat`, `py-readability-metrics`, `pyphen` для слогоделения, нормальная
    сегментация/токенизация). Но L1 — самый дешёвый и неблокирующий слой
    («порог, а не приговор»). Для русского слоги ≈ гласные (точно на Go); для
    английского FK — простая арифметика, эвристика слогов даёт ~85–90%, чего для
    порога достаточно.
  - **L2 (стиль)** шеллит во внешние инструменты при любом host-языке (Vale сам
    на Go, markdownlint на Node) — язык ядра нерелевантен.
  - **L3/L4/L6-механика** — обход ФС, заголовки, манифесты, grep — тривиальны на Go.
- **Консистентность набора:** один тулчейн/CI/идиомы на все четыре репо RRA,
  Go-flavored скиллы (`go test`, godog, формула юнит-тестов), общий контракт
  `DocsAuditor` с `rra-audit-repo`, поставка одним статическим бинарём (удобно
  для CI/pre-commit). Python разорвал бы это и потребовал адаптации скиллов.

Рассмотренные альтернативы:

- **Python целиком** — лучший локальный NLP, но разрыв с дисциплиной/тулчейном и
  runtime-зависимость; оправдан только при глубоком локальном NLP (см. триггер).
- **TypeScript** — хорош для markdown-AST (remark/unified), но readability-библиотеки
  менее зрелые, чем в Python, и тот же разрыв с набором.

## Решение

Реализуем `rra-docs-another` на **Go** — ради единого набора RRA, поскольку
доменный проигрыш локализован в дешёвом неблокирующем L1, а тяжёлая семантика
делегирована LLM (язык-агностично).

Формулы читаемости L1 (Flesch-Kincaid, Оборнева) реализуем нативно на Go;
`textstat` остаётся опциональной сверкой при разработке, не runtime-зависимостью.

**Страховка (не вводится заранее):** если точность L1 на английском окажется
недостаточной — выносим readability в Python-сайдкар через тот же паттерн
автономного I/O-объекта, что и `LinterRunner` (`ReadabilityRunner`, subprocess в
`textstat`/`pyphen`). Go остаётся хостом; Python подключается ровно там, где
лучше, и только по факту.

## Последствия

**Проще:**
- Один язык/тулчейн/CI и общая дисциплина на все четыре репо RRA.
- Поставка одним статическим бинарём; общий контракт `DocsAuditor` с `rra-audit-repo`.

**Сложнее:**
- L1 на английском: нативное слогоделение/сегментация менее точны, чем
  Python-библиотеки (приемлемо для порогового слоя).
- При вводе `ReadabilityRunner` появится runtime-зависимость Python (которую
  концепт изначально хотел избежать) — но только если страховка сработает.

**Триггер пересмотра:** если роадмап потребует глубокого локального NLP без LLM
(когезия Coh-Metrix/TAACO, эмбеддинги, морфология, мультиязычность) — вернуться к
выбору и рассмотреть Python целиком.
