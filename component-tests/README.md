# Component Tests — rra-docs-another

Компонентные тесты CLI-тула как чёрного ящика, в полной изоляции: бинарь-SUT и
LLM-заглушка поднимаются в Docker Compose, godog гоняет настоящий бинарь против
файловых фикстур, без in-code моков. Контракт — `../api-specification/cli.md` +
`../api-specification/report.schema.json`; методология —
[`../skills/component-tests/SKILL.md`](../skills/component-tests/SKILL.md).

Это отдельный Go-модуль (`go.mod`): godog и прочие тест-зависимости не текут в
модуль продукта (он остаётся без внешних зависимостей).

## Требования

- `docker` 24+ с `docker compose` v2.
- Go на хосте **не нужен** — раннер в контейнере.

## Запуск

```bash
./scripts/run-tests.sh                       # smoke + реализованные сценарии
GODOG_TAGS='~@never' ./scripts/run-tests.sh  # всё, включая @wip (красное до S1–S7)
```

Скрипт собирает образы, поднимает `llm-stub`, ждёт его health, запускает раннер,
возвращает его exit-код и гарантированно сносит контейнеры/volumes.

## Структура

```
component-tests/
├── go.mod / go.sum            модуль раннера (godog)
├── Dockerfile.runtime         образ раннера: сборка бинаря-SUT + go test
├── docker-compose.test.yml    стек: llm-stub + tester
├── compose/llm-stub.Dockerfile образ заглушки LLM
├── llm-stub/main.go           OpenAI-совместимая заглушка (стдлиб, Go)
├── scripts/run-tests.sh       точка входа + trap cleanup
├── features/                  .feature: smoke (зелёный) + по подкоманде (@wip)
├── steps/                     godog-степы
│   ├── main_test.go           раннер (по умолчанию пропускает @wip)
│   ├── world.go               World + lifecycle + exec бинаря + разбор отчёта
│   └── cli_steps.go           степы: запуск, код возврата, поля отчёта, error.code
└── testdata/                  фикстуры repo-good / repo-bad
```

## Статус сценариев

- `smoke.feature` — **зелёный** уже на каркасе E0 (проверяет обвязку).
- Остальные `.feature` помечены `@wip` и по умолчанию **пропускаются** (main
  зелёный). Это исполняемая спецификация: каждый слайс S1–S7 снимает `@wip` со
  своей подкоманды, когда реализует её. Гейт E2 опирается на них как на источник
  истины при обратной сверке дизайна.

## Заглушка LLM

Единственный долгоживущий сервис. SQLite-аналога «реальной зависимости с отказом
средой» у LLM нет — это внешний недетерминированный сетевой API, который нельзя
сделать реальным в тесте (не вызвать 429 по требованию, деньги, дрейф). Поэтому
ставим управляемый стенд-ин по реальному протоколу. Режим переключается степом
`LLM-стаб в режиме "<mode>"` до запуска бинаря:

| Режим | Ответ заглушки | Ожидаемый `error.code` |
|---|---|---|
| `healthy` | 200 + verdict | — (happy) |
| `rate_limited` | 429 + Retry-After | `llm_rate_limited` |
| `unavailable` | 503 | `llm_unavailable` |
| `budget_exceeded` | 200 + огромный usage | `llm_budget_exceeded` |

## Доступные степы

**Запуск**
- `запускаю бинарь с аргументами "<args>"` — произвольный запуск (smoke).
- `запускаю "<команда>" на репозитории "<fixture>"` — подкоманда против фикстуры,
  `--format json`; для `fitness`/`assess`/`drift` добавляет `--llm-*` на стаб.
- `LLM-стаб в режиме "<mode>"` — переключить заглушку.

**Проверки**
- `код возврата <n>`
- `отчёт содержит JSON-поле "<путь>" со значением "<v>"` — точное равенство (dotted-путь).
- `отчёт содержит непустое JSON-поле "<путь>"`
- `отчёт содержит JSON-поле "<путь>"` — просто присутствует.
- `в errors[] есть error.code "<code>"`
- `stderr содержит "<подстрока>"`

## Не запускайте `go test` с хоста

Раннер в контейнере — требование изоляции (см. `../AGENTS.md`). Локальный
`go test ./steps/...` даст разное поведение в CI и у разработчика (нет бинаря,
нет стаба, нет фикстур в ожидаемых путях). Всегда через `./scripts/run-tests.sh`.
